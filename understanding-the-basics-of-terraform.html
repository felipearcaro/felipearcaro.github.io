What is Terraform?

Before we jump into coding things out, let's go over some key concepts.

Terraform is a tool used in the field of infrastructure as code (IaC) to create and manage your computer resources, such as servers, networks, and databases. It allows you to define your desired infrastructure in a configuration file using a simple and declarative language.

Once you have defined your infrastructure configuration, Terraform takes care of provisioning and managing the resources for you. It interacts with cloud providers, such as Amazon Web Services (AWS) or Microsoft Azure, to create and update your infrastructure based on the configuration you provided.

The main advantage of using Terraform is that it enables you to treat your infrastructure as code. This means you can version control your infrastructure configuration, collaborate with others, and easily replicate or rebuild your infrastructure in a consistent and repeatable manner.

And since we're going to be provisioning infrastructure in AWS, it's important also important to understand the difference between a source, an instance, and a service.

  • A resource is a logical entity that represents a piece of AWS infrastructure – an S3 bucket, an EC2 instance, an RDS database
  • An instance is a specific occurrence of a resource that is created from a particular template or configuration – an EC2 instance is a virtual server that is launched from an Amazon Machine Image (AMI). Each instance is a unique copy of the AMI, with its own IP address, storage, and other attributes
  • A service is a collection of related AWS resources and API operations that work together to perform a specific function – AWS offers over 200 different services, including compute, storage, database, analytics, machine learning, and more.

Wait, what is this backend thing?

To understand what the Terraform backend is, let's quickly cover what a Terraform state is first.

You probably saw some state-related code in the code snippet I provided earlier:

# Define where the state will be stored
terraform {
  backend "local" {
    path = ".tfstate"
  }
}

Once a Terraform project is started, the state of the managed infrastructure and configuration needs to be stored somewhere. The state is used by Terraform essentially to map real-world resources to my configuration and keep track of metadata.

Here are some of Terraform state's key aspects:

  • Its primary purpose is to store bindings between objects in a remote system (e.g. an EC2 instance) and resource instances declared in a configuration file
  • It's generally saved in a file called terraform.tfstate
  • It can be stored locally as you saw in the boilerplate code I provided earlier. This is recommended for developing purposes
  • It can be stored in the cloud – e.g. in a s3 bucket. This is recommended for deployment purposes, so it can be versioned, encrypted, and securely shared

The backend simply defines where Terraform stores its state data files (local or remote).

How to "run" terraform code

Before I jump into actually creating the AWS resources, I want to quickly go over some very important terraform commands:

  • terraform plan
  • terraform apply
  • terraform destroy
  • terraform show
  • terraform import

Terraform plan

The terraform plan command, as the name states, creates a plan. It can be used to preview the planned changes to the infrastructure.

Under the hood, the command:

  • Reads the current state of any remote objects to make sure that the Terraform state is up-to-date
  • Compares the current configuration to the prior state and notes any differences
  • Proposes a set of change actions that should, if applied, make the remote objects match the configuration

Terraform apply

The terraform plan command, as the name states, applies the changes to the infrastructure. terraform apply can be run without running terraform plan first although it's not recommended.

Terraform destroy

The terraform destroy command can be used to destroy all objects in the infrastructure. terraform plan -destroy can be used to see what destruction will be made beforehand.

It is also worth mentioning that if the infrastructure has already been provisioned with terraform apply, removing objects from the Terraform code and running terraform apply will also destroy them.

Terraform show

The terraform show command can be used at any time to inspect the configuration of a Terraform state or plan.

Terraform import

The terraform import command can be used for bringing existing resources under Terraform management.

Creating an RDS instance

First, I created a free RDS Postgres instance through the Management Console. Since this was the first AWS service on this account, it automatically created a VPC 3 and a Security Group 4 to ensure the instance was securely isolated and protected by a firewall.

To see if the database was working, I used PGAdmin, but I also needed to make sure of a few things:

  • Make the RDS instance publicly available
  • Added my personal IP to the Security Group's inbound rules used by the database so I could connect to it from outside the VPC:
    • Although not recommended, I could have added all ports 0-655353 and all IPs 0.0.0.0/0 to the inbound rules

Everything seems to be working as expected.

Now, I will use terraform import to import it to my terraform file so I can manage it from there.

First, I have to create the most basic version of the aws_db_instance resource on my Terraform file:

resource "aws_db_instance" "my_project_db" {
  engine               = "postgres"
  instance_class       = "db.t2.micro"
  username             = "sidewalk"
  password             = "${var.postgres_pw}"
}

You probably noticed I am using a variable instead of writing my password in plain text. Here's the easiest way to do it:

  • Create a file called vars.tf in the same folder as the main.tf file:
variable "super_secret_password" {
  default = "ThisPassword123!"
}

Call it from your main terraform code like this ${var.super_secret_password}

Then I run terraform import aws_instance.my-project-db <db-instance-id> - and it said it was imported successfully.

There was a problem though - the database I created through the admin console had the parameter instance_class = db.t3.micro while my terraform code said instance_class = db.t2.micro so, when I ran terraform plan it says that particular resource must be replaced.

Then, when I ran terraform apply, the database I created through the Management Console was deleted and I got an error saying there were some important parameters missing to create the new one.

Key learnings on using terraform import:

  • To import an existing infrastructure from AWS, you must have the resource in the terraform code with all required parameters
  • Make sure the configuration you have on your terraform code matches what you have on AWS
  • terraform plan won't show if there are required parameters missing, only terraform apply

Now that I know how this works, I'll create the database directly from my terraform code using AWS documentation on aws_db_instance

resource "aws_db_instance" "my_project_db" {
  engine               = "postgres"
  instance_class       = "db.t3.micro"
  username             = "sidewalk"
  password             = "${var.postgres_pw}"
  allocated_storage    = 20
}

I ran terraform plan to make sure everything was looking good and terraform apply to create the infrastructure. After my database was created, I typed terraform show to see my infrastructure current state – the VPC, the Security Group, etc.

Key learnings on creating infrastructure using Terraform code:

  • It automatically created it on the only VPC available on that AWS account
  • It automatically assigned the RDS instance to the only Security Group available on that AWS account

And here's the updated Terraform code to make the RDS instance publicly accessible:

# RDS instance
resource "aws_db_instance" "my_project_db" {

  instance_class       = "db.t3.micro"

Managing access

Now that my infrastructure is ready, I want to create an IAM 5 user with specific (and restricted) access to use the infrastructure. Here are the steps I will need to go through:

  • Create a policy
  • Create an IAM user
  • Attach the policy to the user

Let's start creating the policies:

data "aws_iam_policy_document" "my_project_policy"{
    statement {
        resources = [aws_instance.my_project_ec2.arn]
        actions = ["ec2:*"]
        effect = "Allow" # Default is Allow
    }

    statement {
        resources = ["aws_db_instance.my_project_db.arn"]
        actions = ["rds:DescribeDBInstances",
                   "rds:DescribeDBClusters",
                   "rds:DescribeGlobalClusters"]
    }             
}

Now we create the user:

# Provides an IAM user
resource "aws_iam_user" "my_project_user" {
  name = "my-project-user"
}

And attach those policies to the user:

# Provides an IAM policy attached to a user
resource "aws_iam_user_policy" "my_project_user_policy"{
    name = "my_project_policy"
    user = aws_iam_user.my_project_user.name
    policy = data.aws_iam_policy_document.my_project_user_policy.json
}

  4. A Security Group is a virtual firewall that controls the inbound and outbound traffic for one or more instances. Every instance within a VPC in AWS must be associated with at least one security group. ↩

  5. IAM - Identity and Access Management is a service that allows you to manage access to AWS resources securely. IAM enables you to create and manage AWS users and groups and control their level of access to AWS resources. ↩
